# 고객가치 프록시 리포트 (스토리 1-2)

**작성일**: 2026-07-20
**대상 함수**: `crm/segment/value.py::customer_value(df)`
**데이터**: `data/bankchurners.parquet` (1-1b가 확보한 실데이터, n=10,127)
**근거 규정**: AD-11(가치 단일 정의·원척도 보존), SPEC CAP-5(2026-07-20 개정본), NFR1(가정 라벨링), NFR3(통화 무단위)

> **이 리포트는 세션 산출물이다.** 준비도 점검 M3에서 확인된 대로 `customer_value()`는 순수 함수이지 파이프라인 단계가 아니다. AC2가 말하는 "산출 리포트"는 이 문서이며, `pipelines/` 단계가 만드는 파일 출력이 아니다. 이 스토리는 파이프라인 파일을 만들지 않는다.

---

## 1. 정의

고객가치는 `Total_Trans_Amt`(연간 총 거래액) **단일 컬럼**으로 정의하며, **원척도 그대로** 반환한다. 정규화·로그 변환·재가중은 이 함수에서 수행하지 않는다.

```
customer_value(df) -> Series[float]   # 인덱스는 입력 프레임 보존
```

`Total_Trans_Amt`의 실데이터 dtype은 `int64`이므로 함수가 **명시적으로 `float`로 캐스팅**한다. 반환 dtype이 입력 데이터의 우연에 의존하지 않게 하기 위함이다.

**단위 없음(NFR3).** BankChurners는 통화 단위를 명시하지 않는다. 위 수치 어디에도 `$`·`₩` 등 통화 기호를 붙이지 않았으며, 하위 스토리·마트·대시보드에서도 붙여서는 안 된다. (P1 3-4에서 통화 기호를 임의로 붙여 오기 사고가 발생했다.)

**분포 관찰**: 평균(4,404)이 중앙값(3,899)보다 크고 최댓값이 중앙값의 4.7배로, 오른쪽 꼬리가 긴 우편향 분포다. 상위 10% 고객이 전체 거래액의 29.8%를 차지한다. 이 편향은 3-1 분면 판정에서 **경계값 선택(중앙값 vs 평균)에 따라 분면 인원이 크게 달라진다**는 뜻이므로, 3-1은 사용한 기준을 명시해야 한다. 로그 변환이 필요하다면(0 이하 값이 0건이라 적용 자체는 가능) **3-1 내부에서** 수행한다 — 이 함수는 변환하지 않는다.

## 3. 가정 (NFR1: 측정이 아니라 가정)

> **[가정 1] 연간 거래액은 수수료 수익의 1차 동인이다.**
> 카드 사업의 인터체인지·수수료 수익은 거래액에 비례하는 구조이며, `Total_Trans_Amt`는 BankChurners가 담고 있는 **유일한 직접 측정 지출량**이다. 따라서 고객가치의 대리변수로 채택한다.

> **[가정 2] 이 축은 수익성이 아니라 거래 규모를 재는 것이다.**
> 아래 한계를 감수하고 사용한다는 것 자체가 가정이다.

### 한계 (모두 가정의 대가로 감수하는 것)

1. **수익성 ≠ 거래액.** 조달비용·연체·대손·고객별 리워드 지급률이 데이터에 없다. 거래액이 큰 고객이 반드시 수익성이 높은 고객은 아니다.
2. **보조 지표 미반영.** `Total_Revolving_Bal`(리볼빙 잔액)과 `Credit_Limit`(한도)은 가치 축에 **산입하지 않는다.** 이는 SPEC CAP-5의 2026-07-20 개정(커밋 `696ffdb`, 준비도 M1 해소)에 따른 것으로, 원안의 "보조 지표 활용" 문구가 AD-11의 단일 정의 원칙과 충돌해 삭제됐다. 두 컬럼은 **CAP-1 세그먼트 프로파일 서술용 참고 지표**로만 쓴다. 리볼빙 잔액은 이자 수익과 관련되므로 이를 배제한 것은 실질적 한계이며, 의도적으로 감수한 단순화다.
3. **기간 정보 부재.** 거래액이 어느 기간의 누적인지 데이터셋이 명시하지 않는다. 고객 간 비교는 모든 고객이 동일 기간을 관측했다는 가정 위에 서 있다.
4. **미래가치 아님.** 이것은 관측된 과거 거래액이지 생애가치(LTV)가 아니다. LTV는 별도 레인(에픽 2, Online Retail 데이터)에서 다루며 AD-1에 따라 두 레인은 만나지 않는다.

## 4. 단일 정의의 기계적 강제 (AC3)

가치 정의가 문서상의 약속으로만 남으면 3-1·3-2·3-3·4-1이 각자 `Total_Trans_Amt`를 집어 쓰는 것을 막을 수 없다. AD-11이 존재하는 이유가 정확히 그 시나리오다.

그래서 구조 가드를 추가했다:

- `tests/structure/checkers.py::find_value_recomputation_violations`
- `crm/` 아래에서 `crm/segment/value.py`를 **제외한** 모든 모듈이 `Total_Trans_Amt`를 참조하면 실패한다.
- **AST 기반**이며 두 표기를 모두 잡는다: `df["Total_Trans_Amt"]`(`ast.Constant`), `df.Total_Trans_Amt`(`ast.Attribute`).
- 텍스트 grep을 쓰지 않은 이유: 주석·docstring의 언급까지 잡아 **규칙을 설명하는 모듈을 규칙 위반으로 보고**한다. 산문에 반응하는 가드는 사람들이 무시하게 만든다.

**자기검증 픽스처 5건**(1-1a 관례)으로 가드가 실제로 문다는 것을 실증했다: 서브스크립트 검출, 어트리뷰트 검출, `value.py` 면제, 산문 오탐 없음, 정상 소비 패턴(`customer_value()` 호출) 통과.

현재 커버리지: **10개 파일 스캔, 위반 0건**. (`crm/` 파이썬 파일 11개에서 면제 대상 `value.py` 1개를 뺀 수. 대상 0건으로 조용히 통과하는 상태가 아니다.)

### 가치 정의를 바꾸려면

`value.py` 하나만 고치면 하위 모든 숫자의 문서상 의미가 조용히 틀어진다. 다음 셋을 **함께** 고쳐야 한다:

1. `crm/segment/value.py::customer_value`
2. 가치 컬럼을 서술하는 마트 스키마 문서
3. SPEC CAP-5의 한계 문구

이 절차는 함수 docstring에도 못박아 두었다.

## 6. 인계 — 1-3·1-6으로 넘기는 누수 경고 (본 스토리 소관 아님)

실데이터 실측 결과, BankChurners에는 **타깃 누수 컬럼 2개**가 있다:

```
Naive_Bayes_Classifier_..._1   Attrition_Flag 상관  +1.0000
Naive_Bayes_Classifier_..._2   Attrition_Flag 상관  -1.0000
(참고) Total_Trans_Amt         Attrition_Flag 상관  -0.1686
```

두 컬럼은 타깃으로 사전 학습된 분류기의 출력이며 타깃과 완전상관이다. 피처로 넣으면 이탈 모델 AUC가 1.0에 붙고 프로젝트 전체가 무의미해진다. Kaggle 데이터셋 설명도 삭제를 지시한다.

- **1-3(피처)**: 피처 집합에서 반드시 배제할 것.
- **1-6(이탈 모델)**: 누수 감사에서 재확인할 것.

**본 스토리에는 영향 없다.** `value.py`는 `Total_Trans_Amt`만 읽으므로 두 컬럼을 건드리지 않으며, 이번 변경에서도 손대지 않았다.
