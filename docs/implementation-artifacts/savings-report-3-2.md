# 캠페인 기대 절감액 리포트 (스토리 3-2, CAP-6 / FR12)

- 산출일: 2026-07-22 · 테스트 **376 passed**(코드리뷰 반영 후)
- 대상: BankChurners 10,127행 (`data/churn_scored.parquet`, `artifact_id 9e1a4d71800f`, 예측자 8개)

### 재현 절차

```python
raw   = pd.read_parquet("data/bankchurners.parquet").set_index("CLIENTNUM")
sc    = pd.read_parquet("data/churn_scored.parquet").set_index("CLIENTNUM")
val   = customer_value(raw).reindex(sc.index)          # crm.segment.value
sav   = expected_saving(sc["churn_prob_calibrated"], val)   # crm.campaign.simulate
quad  = assign_quadrant(sc["churn_score"], val).labels      # crm.campaign.matrix
sav.groupby(quad).agg(["size", "mean", "sum"])
```
- 확률 기준: **`churn_prob_calibrated`**(Platt 보정, 평균 0.1607 = 실제 이탈률). 순위용
  `churn_score`가 아니다 — 아래 「컬럼 선택의 대가」 참조
- 재현: 금액 자체는 커밋·테스트된 `crm/campaign/simulate.py::expected_saving()`가 만든다. **집계
  경로(분면 groupby·합계·비율)는 커밋되지 않았다** — 이 함수는 Series 2개를 받는 순수 함수이고,
  리포에서 이를 호출하는 커밋된 코드는 테스트뿐이다. 아래 「재현 절차」의 6줄로 재구성된다
  (코드리뷰 지적: "커밋·테스트된 함수가 만든다"는 초판 문구는 집계까지 포함하는 것처럼 읽혔다)
- 커밋 포인터: `data_artifact_commit` `3b32749`(`churn_scored.parquet` 생산) ·
  `implementation_commit` = 이 문서와 같은 커밋

## 산식과 가정

```
기대 절감액 = P(이탈) × 고객가치 × 리텐션 성공률 − 캠페인 비용
```

| 입력 | 값 | 성격 |
|---|---|---|
| `P(이탈)` | `churn_prob_calibrated` | **실측**(모델 산출, 보정됨) |
| 고객가치 | `customer_value()` = `Total_Trans_Amt` | **실측**(원척도, 무단위) |
| 리텐션 성공률 | `0.30` | **[가정]** `crm/config.py` `# source: 정책가정` |
| 캠페인 비용 | `5.0` | **[가정]** 무단위. 통화 단위를 붙이지 않는다(NFR3) |

**무단위 표기**: BankChurners는 통화 단위를 갖지 않는다. 이 문서의 모든 금액에 원·달러 기호가 없는
것은 누락이 아니라 규약이다(P1 3-4가 통화를 섞어 실패한 선례).

## 전체 분포

```
양수 8,587 / 10,127 (84.8%)   ← [가정 cost=5.0의 산물] · §② 참조
음수 1,540
합계 1,454,088 | 중앙값 2.83 | 최대 2,932.3 | 최소 −3.25
부호 전환 경계: P(이탈) × 고객가치 = 비용 / 성공률 = 16.6667
```

> **이 블록만 뜯어서 인용하지 말 것.** 84.8%는 발견이 아니라 `비용 = 5.0` 가정의 산물이다.
> 같은 데이터에서 `비용 = 10.0`이면 **37.6%**, `비용 = 20.0`이면 **30.5%**가 된다(아래 §② 표).

## 🚨 이 리포트가 오독되기 쉬운 지점 두 곳

### ① 총합은 2×2를 반박하지 않는다 — 인원수를 보고 있을 뿐이다

| 분면 | n | **1인당 평균** | 총합 | 양수 비율 |
|---|---|---|---|---|
| **Save 우선** | 443 | **1,415.00** | 626,845 | 100.0% |
| 관망 | 2,089 | 382.81 | **799,683** | 100.0% |
| 저비용 유지 | 4,624 | 6.14 | 28,398 | 100.0% |
| 이탈 수용 | 2,971 | −0.28 | −838 | 48.2% |

**관망의 총합(799,683)이 Save 우선(626,845)보다 크다.** 이것을 *"그럼 관망에 예산을 쓰는 게
낫지 않나"* 로 읽으면 안 된다. 관망의 인원이 **4.7배**이기 때문에 생긴 값이다.

예산 제약 하에서 의미 있는 것은 **1인당 기대 절감액**이고, 그 기준으로 Save 우선은 관망의
**3.70배**다. 총합은 *"그 분면 전원을 접촉했을 때"*의 값이므로 **예산이 무한할 때만** 의미가 있다.

그리고 이 리포트는 **누구를 접촉할지 정하지 않는다.** 우선순위는 분면 단위가 아니라 **개인 단위**로
매겨지며, 그것은 `target_priority`를 만드는 **스토리 3-3의 일**이다(AD-12). 여기서 답한 것은
"이 고객 한 명을 접촉하면 얼마인가" 하나뿐이다.

### ② "84.8%가 양수"는 발견이 아니라 `비용 = 5.0`의 산물이다

부호 전환 경계가 `16.6667`이고 고객가치 중앙값이 `3,899.0`이므로, **이탈 확률이 0.0043(0.43%)만
넘으면** 기대 절감액이 양수가 된다. 즉 이 84.8%는 **비용 가정이 낮아서 나온 숫자**이지
"거의 모든 고객에게 캠페인이 이득"이라는 발견이 아니다.

**`비용 = 5.0`에서는** 부호 전환이 분면 하나 안에서만 일어난다 — Save 우선·관망·저비용 유지는
100% 양수이고, 음수 1,540명은 전부 이탈 수용 분면(48.2%만 양수) 안에 있다.

**그러나 이것도 구조가 아니라 가정의 산물이다.** `COST_GRID` 전체를 쓸어본 실측:

| 비용 | Save 우선 | 관망 | 저비용 유지 | 이탈 수용 | 전체 양수 |
|---|---|---|---|---|---|
| 1.0 | 100.0% | 100.0% | 100.0% | 100.0% | 100.0% |
| 2.5 | 100.0% | 100.0% | 100.0% | 87.3% | 96.3% |
| **5.0 (기본)** | 100.0% | 100.0% | 100.0% | 48.2% | **84.8%** |
| 10.0 | 100.0% | 97.2% | **27.9%** | 1.4% | 37.6% |
| 20.0 | 99.5% | 89.5% | 16.8% | 0.0% | 30.5% |

`비용 = 10.0`부터 경계선은 **저비용 유지와 관망을 가로지른다.** 즉 "경계가 맨 아래 칸 안에만
있다"는 서술은 `비용 = 5.0`에서만 참이다. 초판은 이것을 무조건적 구조 사실로 적었고 코드리뷰가
**"84.8%는 가정의 산물이라고 반박하면서 똑같이 가정 의존적인 진술을 사실로 제시했다"**고
지적했다 — 함정 4가 경고한 바로 그 패턴이다.

**비용 가정이 바뀌면 이 그림이 어디서 뒤집히는가**를 답하는 것은 **스토리 3-4(CAP-7)**의 일이다.
`COST_GRID = (1.0, 2.5, 5.0, 10.0, 20.0)`이 이미 정의돼 있고, 대표값 `5.0`이 그 그리드 위에
있음이 import 시점에 검증된다(AD-4) — 그래서 이 리포트의 헤드라인은 **항상 민감도 곡선 위에**
있다. 3-2는 그 입력을 만들 뿐이다.

## 컬럼 선택의 대가 (스토리 3-0이 만든 분리)

같은 산식에 `churn_score`(raw OOF, 순위 전용)를 넣으면:

```
churn_prob_calibrated (올바름) : 합계 1,454,088
churn_score           (틀림)   : 합계 1,730,042   (+19.0%)
```

두 컬럼 모두 `[0, 1]` 범위의 float이고 같은 프레임에 나란히 있다. **타입 시스템도, 산식도, 범위
검사도 이 혼동을 잡지 못한다.** 초판은 이 자리에 계약 테스트가 있다고 적었으나 **그 테스트는
오선택을 잡지 못했다** — 코드리뷰가 실증했고(오선택 호출이 예외 없이 통과), 실제로는 확률 단조성
테스트를 다시 쓴 것이었다.

지금은 **Series 이름을 검사한다**(`_validate_probability_column`). `churn_scored.parquet`에서 컬럼을
그대로 읽으면 `name='churn_score'`가 붙어 있으므로 거부된다.
**가드는 의도적으로 부분적이다** — 산술을 거친 Series는 `name=None`이 되고 이는 통과시킨다. 3-4의
스윕처럼 정당한 호출을 막지 않기 위해서다. 즉 **"파일에서 막 읽은 컬럼"은 잡고 "가공된 값"은
못 잡는다.**

원인: 미보정 OOF 점수의 평균은 `0.1946`인데 실제 이탈률은 `0.1607`이다. 점수를 확률로 읽으면
**계통적으로 부푼다**. 2×2가 raw 점수를 쓰는 것은 **순위만 필요하기 때문**이고, 금액은 숫자의
크기가 의미를 가져야 하므로 보정본을 쓴다.

## 고객가치는 원척도 그대로다 (AD-11)

비용 `5.0`과 고객가치 중앙값 `3,899`는 스케일이 세 자릿수 차이가 난다. **정규화하고 싶어지는
지점이고, 하지 않았다.** 가치를 스케일링하면 산출물이 더 이상 금액이 아니게 되고, 이 컬럼을
소비하는 3-3·4-1이 전부 오염된다. 스케일 차이는 문제가 아니다 — 무단위 규약(NFR3) 아래에서
비용과 가치는 같은 척도 위에 있다.

이 계약은 **함수 안에서만** 테스트가 지킨다: sentinel Series를 넣어 출력이 그 값 그대로 나오는지
검증한다(`test_the_value_input_is_used_exactly_as_given` — 서로 다른 3점을 써서 `value * 0.02`도
`np.log1p(value)`도 KILL된다). **1-2가 넘긴 숙제의 나머지 절반은 열려 있다** — 인계 문구가 겨냥한
것은 `customer_value(df) * 0.02` 같은 **호출부** 재가중인데, `expected_saving`은 `customer_value`를
호출하지 않으므로 monkeypatch할 지점이 없다. 그 자리는 이 함수를 실제로 호출하는 코드(4-1)가
생길 때 닫힌다.

## 액션으로 이어질 때 (회고 A6)

이 리포트는 **금액만** 답한다. "그래서 무엇을 제안할 것인가"의 후보는 **SPEC CAP-3의 예시가 아니라
1-7의 실측 매핑**에서 온다 — SPEC 예시 *"한도 불만형 → 한도 상향"*은 1-7 실측에서 **부정됐다**
(`Avg_Utilization_Ratio`는 **낮을수록** 위험, r=−0.64).

Save 우선 분면(1인당 1,415.00)이 실제로 어떤 세그먼트와 겹치는지는 1-7의 세그먼트 4(저활동
고위험층)가 이미 예고한 바 있다 — *"고비용 오퍼 전에 가치 축과 교차. 3-1의 2×2가 이 세그먼트 내
우선순위를 정한다"*. 이 리포트의 1인당 금액이 그 교차의 결과다.
